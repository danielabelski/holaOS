Memory Structure Implementation Notes

Design decisions

  • Removed the external memory proposal HTTP routes before replacing the backend so proposal review is no longer part of the supported runtime surface.
  • Removed the bridge-layer workspace.memory.* native job handlers as an explicit cut toward a tree-native backend with fewer legacy entry points.
  • Removed the internal memoryService.capture(...) API because it only existed to support the old proactive context capture flow, which is no longer part of the supported memory direction.
  • Removed the proactive workspace context HTTP route and native bridge job path rather than preserving them on top of the new memory work.
  • Rewrote the PRD around one memory root with two ownership categories, interaction and integration, replacing the earlier topic and daily_digests framing.
  • Locked the node model so every persisted node body is markdown, source leaves are bottom-level documents with no children, and summary nodes are markdown documents that summarize either source leaves or lower summary nodes.
  • Added an explicit ingestion vocabulary to the PRD: event is the raw incoming memory signal, chunk is the normalized admission-stage unit, and source leaf is the durable markdown node created only after a chunk is admitted.
  • Locked the topology vocabulary to memory root -> tree category -> tree, and within each tree to root summary node -> intermediate summary nodes -> source leaves.
  • Chose a no-shared-leaves design. Each leaf belongs to exactly one tree, and cross-category recall is handled by the retrieval layer rather than by multi-tree leaf membership.
  • Attached date and time metadata to every node while explicitly keeping date out of the ownership hierarchy for v1.
  • Locked the lifecycle direction to event -> chunk -> source leaf -> summary node, with chunks treated as transient admission-stage units and leaves treated as durable bottom-level nodes.
  • Locked the write-lifecycle preference so admitted source leaves are treated as immutable, and new information should usually create a new leaf with supersession metadata rather than rewriting old leaf bodies in place.
  • Chose an LLM-driven interaction entity assignment path for v1, where the model decides whether a chunk fits an existing entity or needs a new one, while backend code still owns canonical id generation, validation, and fallback routing.
  • Added an explicit interaction entity decision matrix: low admission confidence rejects the chunk, high existing-match confidence attaches to an existing entity, high new-entity confidence mints a new entity, and otherwise the durable leaf falls back to interaction:uncategorized.
  • Simplified phase 1 to interaction-only implementation. Integration trees remain part of the target architecture but are deferred from the first build.
  • Removed hotness from phase 1. An interaction entity becomes a tree as soon as its first admitted leaf is assigned, and summary layers are created only by seal thresholds.
  • Added a concrete phase-1 upsert algorithm covering chunk admission, LLM entity assignment, backend validation, duplicate no-op handling, superseding leaf creation, tree creation, and summary invalidation/rebuild.
  • Added a retrieval design where durable memory access becomes an agent tool backed by the tree query engine, with summary-first traversal used for broad recall and leaf drill-down used for exact evidence.
  • Explicitly chose to embed sealed intermediate summary nodes as well as leaves, using summary embeddings for branch selection and leaf embeddings for final evidence retrieval.
  • Updated phase-1 summary generation so interaction summary nodes use an LLM-authored summary sentence when a summary model client is available, while keeping deterministic node ids, deterministic wrapper markdown, and deterministic fallback behavior when no model client is available.
  • Implemented phase 1 by introducing dedicated interaction-memory tables in the runtime state store for entities, leaves, summary nodes, tree edges, and node embeddings, instead of reusing the legacy memory_entries catalog.
  • Cut runtime durable writeback and bootstrap recall directly over to the new interaction-memory module, so completed turns now persist interaction leaves and runner bootstrap now recalls interaction hits without going through MEMORY.md planning.
  • Added a new runtime agent tool, memory_retrieve, as the supported agent-facing entry point for durable memory retrieval in phase 1.
  • Removed the desktop workspace-ready heartbeat path entirely, including the proactive ingest emit and its earlier runtime-captured context dependency, so workspace creation no longer depends on legacy proactive bootstrap behavior.
  • Removed heuristic durable-memory candidate extraction from turn writeback. Interaction memory now relies on model extraction only, and extraction runs on every completed turn when a model client is available.
  • Changed interaction-memory writeback from per-turn extraction to non-overlapping batches of three completed turns per session. The workspace runtime DB now stores a per-session batch cursor in workspace_runtime_metadata, and the extractor prompt is framed around the current three-turn batch rather than a single turn.
  • Narrowed the runtime prompt contract so AGENTS.md is framed as the workspace-wide defaults and control surface, while contextual facts and subject-specific procedures are framed as interaction-memory recall that should use memory_retrieve.
  • Shifted the memory_retrieve prompt guidance from examples of recall-like question forms to a source-of-truth heuristic: prefer the most local authoritative source, use memory first for workspace-specific or previously learned knowledge, and only broaden to files, integrations, workspace tools, or web if memory does not return a strong relevant result.
  • Narrowed the post-run memory extraction prompt so it asks only for contextual recall memory. The extractor now explicitly excludes workspace-wide defaults, recurring commands, response-style preferences, and default release or verification procedures that should instead be handled through AGENTS.md during the run.
  • Added repo-level memory experiment tooling: a cleanup script for wiping interaction-memory state in a live workspace, a checked-in phase-1 evaluation fixture set, and a live evaluator that drives write turns, reader turns, direct retrieval checks, retrieval-latency measurement, tree-quality inspection, and AGENTS.md delta reporting against the current memory backend.
  • Captured a dedicated batch/materialization reliability plan for the 3-turn interaction writeback path. The plan treats extraction, persistence, and summary rebuild as distinct stages, adds explicit batch-state tracking, and treats context-overflow handling as a first-class extraction fallback concern rather than assuming the clipped prompts are always safe.
  • Implemented the first reliability slice for interaction writeback batches. The runtime now records explicit per-batch state in workspace_runtime_metadata, including running/completed/failed status, timing breakdowns, extraction attempt counts, and whether sub-batch fallback was used.
  • Changed extraction failure semantics so a failed model extraction no longer advances the session batch cursor. Failed batches now remain retryable instead of being silently treated as empty.
  • Added oversized or failed full-batch extraction fallback by splitting a 3-turn batch into smaller extraction units and merging the resulting candidates. This currently uses prompt-size estimation plus retry-on-null behavior, without changing the top-level 3-turn batch abstraction.
  • Implemented the second reliability slice as a per-session in-process writeback lease. Overlapping writeback calls for the same workspace/session now skip duplicate extraction work while a batch is already in flight, and the current lease state is mirrored into workspace_runtime_metadata for visibility.
  • Started phase 2 by adding a dedicated integration tree family. Integration tree identity is now provider + account, persisted in new runtime-state tables for trees, leaves, summary nodes, edges, and embeddings.
  • Added a separate integration-memory.ts backend that mirrors the interaction tree shape but skips entity assignment. Integration leaves now belong directly to one provider/account tree, and summary layers are built above those leaves with the same markdown-node contract.
  • Generalized runtime memory retrieval into a combined workspace-memory layer. memory_retrieve and pre-run recalled memory can now search interaction, integration, or both categories through one wrapper.
  • Added a writeback-side recall filter for interaction batches. Three-turn memory extraction now excludes turns that look like reader/retrieval turns, based on memory_retrieve tool usage plus a question-style user prompt, and the extractor prompt now treats recalled answers as read-only context rather than new memory evidence.
  • Stopped duplicating current-batch user prompts into the extractor's "recent user messages" context. The extractor now gets prior-session user context separately from the current batch, which reduces accidental re-learning from the same batch.
  • Tightened the prompt-only boundary between AGENTS.md and memory without adding backend enforcement. The prompt now frames AGENTS.md as information the agent should obey by default on most future runs even when the current subject is not in scope, and it explicitly routes named-subject contextual knowledge to memory instead.
  • Started the first real integration-tree ingest path with Gmail. Connection activation and a new manual integrations-pane action now trigger a Gmail context fetch workflow that pulls account profile plus recent email metadata and persists them into the global per-account integration tree.
  • Formalized a new runtime route, POST /api/v1/integrations/context-fetch, as the supported backend entry point for integration context harvesting. The older debug/composio-runtime-test route remains as a diagnostic probe, but the product flow now targets the formal route.
  • Aligned integration-memory visibility with the same workspace source of truth used by Settings → Integrations: active connections plus workspace overrides now determine which global integration trees are visible to a workspace. Explicit integration_bindings remain in place for app/broker routing, but no longer gate memory retrieval.
  • Tightened the main-session and base runtime prompt so unknown fact lookups probe memory_retrieve before browser, web, file search, or other broader retrieval routes. Browser and other live external surfaces are now explicitly framed as later fallbacks unless the user is asking for current live state or UI-specific information.
  • Removed raw memory storage paths from the agent-facing recall surfaces. memory_retrieve now returns evidence payloads without file-system paths, and recalled-memory prompt context no longer prints storage locations, so the agent treats memory hits as the answer surface rather than browsing backing markdown files.
  • Added a graph projection to Settings → Memory without changing the storage model. The graph view now exposes a workspace interaction forest and a workspace-visible integration forest as read-only projections over the existing tree tables and markdown bodies.
  • Changed the Memory pane to default to graph view and expanded the graph export so it includes every visible layer in one pass. The UI no longer requires a tree drill-in step just to reveal summaries or leaves.
  • Reworked the Memory pane layout so the explorer stays fixed on the left while the right-hand surface alternates between graph and file views. Selecting a file in the explorer or a graph node now routes the right side into file inspection instead of using a separate inspector column.
  • Simplified the top-level Memory pane mode model so it is always either workspace memory or integration memory. The workspace picker is now only shown in workspace mode, while integration mode hides that selector and uses the current workspace context only for visibility filtering.
  • Replaced the first graph renderer's force-style layout with a tree-aware forest layout. The graph payload is unchanged, but the SVG now positions nodes by parent/child structure so workspace and integration trees render as actual hierarchies instead of one tangled network.
  • Expanded the live evaluator into a phase-2 fixture set with longer turns, denser multi-entity interaction sessions, and a live Gmail integration-fetch scenario. The cleanup helper now wipes both interaction and integration tree state so repeated eval runs start from a true cross-forest baseline.
  • Loosened the evaluator’s interaction-tree assertions so scenarios can resolve expected entities by canonical name and type instead of hardcoding exact entity ids. That keeps the phase-2 fixture stable even when entity slugs or ids drift.
  • Generalized the integration eval runner away from a Gmail-specific scenario kind. Integration context-fetch scenarios now share one generic live-fetch path keyed by provider_id, and the report template context is derived from the fetched leaves so fixtures can reference dynamic repository, notification, or message titles without hardcoding live account contents.
  • Split integration coverage into provider-specific fixtures plus cross-source mixed scenarios. The mixed suite now intentionally asks questions that combine earlier interaction facts with later Gmail or GitHub fetches so the eval can pressure retrieval when both memory families coexist in one run.
  • Fixed the live integration evaluator to read active integration connections from the host control-plane database rather than the workspace runtime database. In this desktop profile, Gmail and GitHub connections are stored under sandbox-host/state/control-plane.db, so the earlier workspace-local lookup produced false integration failures even when the providers were connected.
  • Updated the live evaluator for the newer hierarchical integration graph shape. Cleanup and count snapshots now wipe and inspect integration trees in the control-plane database, and integration scenarios can assert branch keys, entity-key prefixes, and relation types rather than only checking for a profile leaf.
  • Changed the evaluator’s --clean behavior from a one-time pre-suite wipe to per-scenario isolation. Every cleaned scenario now resets interaction memory, integration memory, session-history tables, on-disk session artifacts, and AGENTS.md before the scenario runs so later cases cannot inherit prior session state.
  • Adjusted the integration evaluator to the newer async fetch contract. POST /api/v1/integrations/context-fetch now starts a run and returns a running status snapshot, so the evaluator polls GET /api/v1/integrations/context-fetch until the connection reaches a terminal status before validating the resulting tree.
  • Added a hybrid semantic dedupe layer at the interaction-leaf persistence boundary. Exact fingerprint duplicates still short-circuit cheaply, but near-duplicate active leaves in the same entity now go through a tiny LLM-reviewed shortlist before the backend decides whether to noop, supersede, or create.
  • Changed interaction owner selection to be stable-subject-first instead of memory-kind-first. The classifier now distinguishes the enduring owner (customer, project, system, workflow, etc.) from the memory kind (procedure, contact, threshold, channel, dashboard), so operational or procedural facts about a named subject no longer default to workflow ownership just because they describe a process.

Deviations

  • Left the underlying proposal record types and proposal creation internals in place for now. This change removes the exposed APIs first and avoids coupling the initial cleanup to the larger backend rewrite.
  • Kept /api/v1/memory/sync in place for now even though the current filesystem backend only returns a success wrapper around status. That route can be removed or redefined once the new backend contract is in place.
  • Kept the generic file-oriented FilesystemMemoryService surface for search, get, and upsert so other workspace features that still write markdown artifacts can continue to use it while durable interaction memory lives in the new tree backend.
  • Used deterministic summary-node bodies derived from child titles and summaries for phase 1 instead of invoking an LLM to author tree summaries. This keeps sealing synchronous and testable while the tree structure settles.
  • Replaced the old recall-selection model flow entirely. Runner bootstrap now retrieves directly from interaction leaves and summaries, and the only model call in recall is the optional embedding request.
  • Interaction-memory retrieval currently uses the default recall embedding model (text-embedding-3-small) rather than the runtime background-tasks chat model. That keeps retrieval on a stable embedding model even when the execution model differs.
  • Summary-node bodies are still structurally wrapped by backend templates. The LLM currently supplies the summary prose for each summary node, not the full markdown file.
  • Workspace creation no longer emits the legacy proactive heartbeat at all. This intentionally removes a bootstrap side effect that was only used by the deprecated proactive proposal flow.
  • Interaction durable-memory writeback no longer has deterministic heuristic fallbacks for workspace facts, procedures, or recurring blockers. Those memories now depend on the model extractor deciding they are durable enough to persist.
  • The current prompt still leaves automatic interaction-memory writeback as a backend path rather than an explicit agent write tool, so the prompt change only narrows where the agent should write defaults versus where it should read contextual memory.
  • Fixed the front-session runtime harness so memory_retrieve now routes through the same capability client path as the direct HTTP endpoint. The missing request-plan branch lived in runtime-tool-capability-client.ts.
  • Integration-event routing is still not wired in. This pass only adds the backend tree family, persistence helpers, summary rebuilds, combined retrieval, and direct API tests; no live integration ingest source uses it yet.
  • The initial reader-turn filter is intentionally narrow: it only excludes turns that both used memory_retrieve and look like recall questions. That fixes the observed eval regressions without broadly suppressing question-shaped source turns, but pre-run recalled answers without an explicit memory tool call may still need a second pass.
  • The AGENTS.md leakage fix in this pass is prompt-only. That keeps the control plane simple and avoids overfitting eval terms into backend validators, but it means success still depends on agent compliance and should be rechecked with the live eval harness.
  • The first integration context fetch supports Gmail only. Other providers intentionally return a supported=false result instead of attempting a generic ingest, because Composio response shapes are tool-specific and we have not defined provider-specific normalizers yet.
  • The storage-layer path fields remain in backend records and internal retrieval types for indexing, memory browsing, and tests, but they are intentionally hidden from the runtime tool contract that the agent sees.
  • The new integration eval coverage uses a live Gmail account scenario with skip_if_unavailable semantics instead of seeding deterministic integration trees inside the script. That keeps the evaluator aligned with the real fetch path, but it means the integration slice only runs when an active Gmail connection exists in the target workspace profile.
  • The GitHub eval avoids hardcoded repository or issue names from the live account. Instead it builds answer expectations from the fetched integration leaves at runtime, so the suite verifies behavior against real connected data rather than against a seeded canonical GitHub fixture.

Tradeoffs

  • This keeps the first step small and testable, but it means there is still dormant proposal-related backend code that will need removal in a later pass.
  • The PRD now prefers strict tree ownership over multi-tree projections. That simplifies storage and tree semantics, but it pushes more cross-category composition into the retrieval layer.
  • Keeping the generic file-memory abstraction alongside the new interaction backend makes the transition much less disruptive for unrelated artifact-writing flows, but it leaves the public /api/v1/memory/* routes semantically broader than the tree-native durable backend they now report on.
  • Deterministic summaries avoid another model dependency in the hot path, but they produce flatter summary prose than a dedicated summarization model would.
  • Using an LLM only for the summary prose raises summary quality without making node identity unstable, but it still adds model latency and possible variance whenever tree rebuilds run with a summary model client.
  • Making interaction-memory extraction model-only removes duplicate heuristic/model leaves and makes the tree easier to inspect, but it also increases dependence on extractor prompt quality and model availability for memory growth.
  • Batching writeback in groups of three should reduce duplicate readback learning and give the extractor more context for entity assignment, but it also delays new memory visibility by up to two completed turns and currently tracks progress at the session level rather than by semantic event source.
  • Making the AGENTS.md vs memory distinction explicit should reduce overlap and make retrieval behavior more observable, but it also means the agent now depends more on actually invoking memory_retrieve for contextual recall instead of relying on prompt-preloaded workspace notes.
  • The retrieval surface now works in both places: direct HTTP capability calls and front-session agent tool calls go through the same runtime capability route. That reduces the chance of prompt/tool drift, but it means runtime-tool additions now need harness-client coverage as well as API-route coverage.
  • The evaluation runner uses direct runtime API calls plus runtime-DB inspection rather than a mock harness. That gives a realistic closed-loop experiment and a richer report, but it means the scripts depend on a live local runtime and intentionally operate against real workspace memory state.
  • The integration backend intentionally duplicates the current interaction-tree implementation instead of extracting a shared generic tree core first. That keeps this phase lower-risk and easier to land, but it leaves obvious future refactor room once both categories stabilize.
  • The combined retrieval wrapper currently plans categories conservatively: explicit category requests win, tree-id prefixes narrow the search when present, and otherwise it searches both categories when both have active data. That avoids brittle provenance rules in the backend, but it may do more work than a future smarter planner.
  • Filtering recall-heavy turns before extraction reduces duplicate leaves and keeps the batch model focused on source turns, but it also means the current fix depends on accurate turn metadata. If agent recall increasingly comes from pre-run memory instead of explicit memory_retrieve, we may need a stronger turn classifier later.
  • Making the AGENTS.md rule more abstract around default future behavior should generalize better than enumerating eval-specific customer or project cases, but it may also make the agent slightly more conservative about recording legitimate workspace defaults until the wording is tuned with live runs.
  • The new reliability plan keeps the current branch factor and synchronous rebuild semantics for now. That avoids mixing correctness work with tree-shape tuning, but it means the first pass is focused on preventing silent batch loss and understanding latency rather than immediately optimizing throughput.
  • The first reliability slice stores batch telemetry in generic runtime metadata keys rather than introducing a dedicated batch table. That keeps the schema unchanged for now and makes the change easier to land, but it also means querying batch history will stay more ad hoc until we decide whether this metadata should graduate into a first-class table.
  • The current oversize handling treats any null extraction response as eligible for sub-batch fallback when the batch has more than one included turn. That is conservative and prevents silent loss, but it may retry on transient upstream failures that were not actually caused by prompt size.
  • The new batch lease is intentionally process-local. That is enough for the current single embedded runtime process and avoids introducing cross-process coordination complexity in the same pass, but it would need to become a database-backed compare-and-set lease if writeback ever runs concurrently across multiple runtime processes.
  • The Gmail fetch workflow rebuilds integration summaries deterministically without workspace-scoped summary or embedding models. That keeps the new global account trees independent from any single workspace, but it also means Gmail trees will not get semantic summary prose or embeddings until we define a workspace-independent model-client strategy for global memory.
  • The new semantic dedupe pass is intentionally same-entity only and only runs after a cheap lexical shortlist. That keeps write latency bounded and avoids global over-merging, but it also means semantically duplicated leaves that land under different entities still need better entity resolution rather than relying on dedupe to clean them up later.
  • The stable-subject classifier uses a broad operational-slot lexicon and lightweight subject-type heuristics instead of a fully model-owned owner taxonomy. That keeps owner selection deterministic and cheaper on the hot path, but it means the fallback path will still need periodic tuning as we see new subject shapes in the wild.
  • Added a first-pass Settings → Memory surface as a read-only filesystem-style browser instead of jumping straight to a graph. The MVP shows the logical memory tree and raw markdown bodies, which makes it easier to debug storage correctness before we add higher-level visualization semantics.
  • Refined the Memory settings pane so it is not locked to the shell's current workspace selection. It now has its own workspace picker plus an All / Workspace / Integrations scope switch, which makes it easier to inspect one workspace's interaction tree separately from its visible integration trees.
  • The first graph renderer intentionally uses a minimalist dot-and-edge SVG instead of labeled cards or a graph library. That makes dense forests much more legible, but it also means the graph itself carries almost no inline text and depends on the inspector for detail.
  • Showing every visible node layer in one graph trades compactness for completeness. Dense workspaces can become visually busier, but the user can now inspect full tree structure without hidden expansion states.
  • The phase-2 eval fixture now prefers longer natural-language turns and denser batches over short atomic prompts. That better reflects real usage and exercises extraction clipping, but it also makes live runs slower and more sensitive to batching latency than the original phase-1 fixture.
  • The mixed integration scenarios now seed their own interaction memory inside the same scenario before running the integration fetch. That keeps strict per-scenario cleanup as the default for all future eval runs and avoids cross-scenario contamination, but it also makes mixed scenarios slower because they now pay both the interaction writeback cost and the integration fetch cost in one case.
  • Deterministic workspace onboarding no longer auto-enqueues a starter prompt into the main session after the user clicks Continue. That keeps onboarding focused on connectivity instead of silently injecting a first task, but it also removes a guided “first-run” nudge that may need to come back later as an explicit user-triggered suggestion instead of an automatic queued message.

Open questions

  • The PRD intentionally defers the exact granularity of integration trees, future hotness rules for interaction-tree materialization, the long-term fate of the generic /api/v1/memory/* routes, and whether later phases should upgrade summary generation from deterministic templates to model-written summaries.
  • Model extraction currently runs on every completed turn whenever a model client is available. If that proves too expensive in production, the next decision will be whether to introduce smarter gating or a cheaper extraction model without reintroducing heuristic duplication.
  • Live verification now succeeds end to end for interaction-memory recall: after runtime restart, a fresh main-session agent answered Orchid contact and procedure questions correctly using only memory_retrieve.
  • Phase 2 still needs the real integration ingest path. We now have the account-owned tree backend, but the next decision is which concrete integration event stream lands there first and what subject-key policy those integration leaves should use.
  • Reader-turn re-ingestion is only partially solved in this pass. The new filter should stop the explicit memory_retrieve feedback loop that showed up in eval, but we still need a follow-up live run to see whether pre-run recalled answers without a tool call are being re-learned.
  • The next live eval should verify whether the stronger prompt wording actually stops named-subject leakage into AGENTS.md from a clean baseline, especially in scenarios that previously passed on recall quality but failed on boundary hygiene.
  • The next reliability pass should decide whether batch completion should continue to wait for summary rebuilds or whether leaf persistence and summary materialization need separate completion semantics. The current implementation still advances the cursor only after rebuild completes.
  • The next integration phase needs a durable incremental-sync plan. Gmail profile fetch records historyId in the profile leaf, but this pass still does a simple recent-email snapshot rather than using GMAIL_LIST_HISTORY or per-account cursors.
